### D-009: 실행 모드 정의 확장 + Work Order/Report 폴더 역할 확정

```
결정:
  - "실행 모드(Execution)"는 코드뿐 아니라 문서/폴더/설정 등 리포지토리에 쓰기가 발생하는 모든 행위를 포함한다.
  - 업무지시서는 `docs/work_orders/`에 보관하고, 보고서/분석 산출물은 `docs/reports/`에 보관한다.
  - `docs/analysis/`는 현재 도입하지 않는다(필요 시 디렉터 승인 후 결정 기록).
이유:
  - PM의 독단적 폴더 생성/마스터 문서 수정 사고 재발 방지(승인 관문을 명확히 하기 위함)
  - 지시(Work Order)와 결과(Report)를 분리해 추적성과 검색성을 확보
  - 신규 폴더 남발로 인한 문서 스프롤 방지
대안:
  - 문서/폴더 변경은 실행 모드에서 제외(권한 오판 가능성 증가)
  - Work Order와 Report를 동일 폴더에 혼재(권위/역할 혼선 증가)
  - `docs/analysis/`를 즉시 도입(폴더 증식/경계 중복 위험)
변경 가능 여부: 중간 (디렉터 승인 후 변경 가능)
일시: 2026-02-13
```

### D-010: 세션 시작 Quick/Conditional/Full Start 도입 (자동 주입 최적화)

```
결정:
  - 세션 시작 기본은 Quick Start로 운영한다: `CLAUDE.md` + `04_CURRENT_STATE.md` + `05_NEXT_TASK.md`.
  - 작업 유형에 따라 `01/02/03`을 조건부(Conditional Load)로 추가 로드한다.
  - 거버넌스/정본/운영 프로토콜 파일을 수정하거나 디렉터가 요청하면 Full Start로 `01~05` 전체를 재확인한다.
  - Codex 프로젝트 `config.toml`의 `priority_files`는 Quick Start 세트로 유지한다.
이유:
  - 매 세션마다 `01~05` 전체를 강제 주입하면 불필요한 재독 비용이 누적되어 작업 퍼포먼스/비용 효율이 저하됨
  - Quick로 시작하되, 조건부 로드/풀스타트 트리거로 오판 리스크를 관리하는 편이 균형이 좋음
대안:
  - 항상 `01~05` 전체 로드(안전하지만 비용 증가)
  - 사용자가 매번 문서 로드를 지시(운영 피로 증가)
변경 가능 여부: 높음 (효율/오판 리스크에 따라 조정)
일시: 2026-02-13
```
